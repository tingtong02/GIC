# GIC 项目静态网页展示内容要求（Requirements v1）

## 1. 文档目的

本文档用于定义一个**静态网页**在展示 GIC 项目时应包含的内容范围、信息组织方式、叙事重点与文字类型要求。  
它不是视觉设计稿，也不是前端技术实现说明，而是**内容与信息架构要求文档**。

目标是让静态网页能够同时服务于以下几类读者：

- 初次接触项目的评审者或老师
- 想快速理解项目价值与边界的研究者
- 想接手仓库或继续开发的工程实现者
- 想看结果、案例与结论而不想先读完整仓库的人

---

## 2. 展示目标

网页的核心目标不是“把仓库内容搬上去”，而是清楚回答以下问题：

1. 这个项目要解决什么问题？
2. 为什么这个问题难？
3. 项目采用了怎样的技术路线？
4. 每一阶段做了什么，为什么要这么拆？
5. 当前默认方案是什么，为什么不是别的方案？
6. KG 在项目里是什么地位，为什么是可选而不是默认？
7. 项目当前最重要的结果是什么？
8. 项目当前最重要的限制是什么？
9. 如果后续继续开发，最优先应做什么？

---

## 3. 整体内容结构要求

建议网页内容至少组织为以下 10 个内容区块：

1. 项目总览
2. 问题背景与研究动机
3. 技术路线总览
4. 分阶段演进过程
5. 当前系统架构
6. 结果与对比
7. 真实事件验证与证据边界
8. 工程实现与仓库体系
9. 关键 trade-off 与设计取舍
10. 当前限制与未来工作

每个区块都必须承担明确的信息功能，不能只是“放一些图和几段说明”。

---

## 4. 首页 / 首屏必须展示的内容

首页第一屏必须让读者在极短时间内知道：

- 项目名称
- 一句简明的项目定位
- 当前系统默认路径
- 当前可选增强路径
- 当前项目所处状态（如：已完成端到端研究系统构建）
- 一个简短但真实的结果摘要
- 一个明确的限制提醒

### 4.1 推荐包含的信息点

- 项目全称 / 简称
- 一句话概述，例如：
  - 基于物理先验、图学习与知识增强的 GIC 网络级态势重建系统
- 当前默认路径：
  - Phase 5 frozen default without KG
- 当前可选路径：
  - Phase 6 `feature_only` with KG
- 当前状态：
  - 已形成 Phase 0–Phase 8 的完整研究系统
- 一条代表性结论：
  - 默认主模型在受控 benchmark 上显著优于早期图基线
- 一条限制结论：
  - 真实事件验证仍基于保守、局部、代理式证据，不能视为全网真实精度结论

### 4.2 首屏文字要求

首页文字必须：

- 简洁但不夸大
- 同时给出“项目价值”和“证据边界”
- 避免使用营销式口号
- 不要一上来堆模型名和公式

---

## 5. 问题背景与研究动机区块要求

这个区块的目标是让读者理解“为什么要做这个项目”，而不是只知道“做了什么模型”。

### 5.1 必须说明的问题

至少要讲清楚：

- GIC 监测与网络级态势重建为什么重要
- 稀疏观测、噪声、真实数据稀缺为什么构成困难
- 为什么单纯静态物理模型不够
- 为什么纯黑箱学习也不够稳妥
- 为什么项目最后走向“物理 + 图学习 + 可选 KG”的组合

### 5.2 推荐展示内容

可以用以下内容组织：

- 一个简短背景说明
- 一个“问题难点列表”
- 一个“传统方案 vs 本项目路线”的对比表
- 一个“从 sparse point 到 network-wide surface”的概念示意

### 5.3 文本风格要求

文字应偏研究型、解释型，而不是教材式长篇背景。  
每段文字应服务于回答“为什么需要这一条技术路线”。

---

## 6. 技术路线总览区块要求

这个区块是网页的主骨架之一。  
应展示从 Phase 0 到 Phase 8 的完整演进路线，但不需要把所有阶段写得一样重。

### 6.1 必须展示的层次

至少应分为以下几层：

- 数据层
- 物理层
- 信号前端层
- 图学习层
- 主模型层
- KG 增强层
- 真实验证层
- 最终系统整合层

### 6.2 建议展示形式

可用：

- 一张总流程图
- 一条阶段时间线 / phase roadmap
- 一个“输入 -> 中间资产 -> 模型 -> 评估 -> final system”的流程示意

### 6.3 每一层要说明的内容

每层至少说明：

- 这一层解决什么问题
- 它的输入来自哪里
- 它的输出给谁用
- 为什么不能省略
- 它在最终系统中是否仍是核心依赖

---

## 7. 分阶段演进过程区块要求

这个区块不是简单列出 Phase 0–Phase 8，而是要展示项目是如何逐步形成当前系统的。

### 7.1 每个阶段至少应包含

- 阶段名称
- 阶段目标
- 关键产出
- 最重要结论
- 对下一阶段的影响

### 7.2 需要特别强调的阶段

建议网页中重点写这些阶段：

- Phase 4：图基线建立，但 graph 并未自动胜过 non-graph baseline
- Phase 5：physics-informed temporal GNN 成为主线
- Phase 6：KG 提供增强但未成为默认路径
- Phase 7：真实事件验证形成保守证据边界
- Phase 8：系统最终固化与交付

### 7.3 必须体现的工程思维

这里不能只讲“成功故事”，还要体现：

- 为什么 Phase 4 的结果反而推动了 Phase 5 的设计
- 为什么 Phase 6 没被删除
- 为什么 Phase 7 不夸大真实世界结论
- 为什么 Phase 8 先做冻结、复现、整合，而不是继续改模型

---

## 8. 当前系统架构区块要求

这个区块应回答：**最终系统现在到底长什么样**。

### 8.1 必须展示的内容

至少包括：

- 默认运行路径（without KG）
- 可选运行路径（with KG / feature_only）
- final config bundle 的作用
- final pipeline 的主要步骤
- 主要输入、主要输出、主要中间资产

### 8.2 应明确区分两条路径

必须清楚区分：

#### 默认路径
- Phase 5 frozen default without KG

#### 可选路径
- Phase 6 `feature_only` with KG

### 8.3 对比方式要求

建议采用对照块的方式说明：

- 默认路径为什么是默认
- 可选路径在哪些方面更强
- 可选路径为什么没有上升为默认
- 两者都应保留的理由是什么

### 8.4 必须写出的结论

要明确告诉读者：

- “默认”不等于“别的路径无价值”
- “可选 KG”不是失败品，而是尚未获得足够证据取代默认路径

---

## 9. 结果与对比区块要求

这是网页中最容易被读者直接关注的部分，因此必须组织得非常清楚。

### 9.1 必须至少分成两类结果

#### A. Synthetic / controlled benchmark 结果
展示：
- Phase 4 baseline
- Phase 5 default
- Phase 6 feature_only

#### B. Real-event validation 结果
展示：
- 当前真实事件 decision
- Phase 5 与 Phase 6 各自的优劣
- 为什么最终 default 仍然是 without KG

### 9.2 结果展示要求

结果区块不能只放一个表。  
建议至少包含：

- 一张总表（各方案主要指标）
- 一段解释文字（告诉读者该怎么读）
- 一个“最终默认决策”的说明块

### 9.3 必须体现的 trade-off

必须明确展示：

- Synthetic benchmark 上，Phase 6 `feature_only` 在部分误差指标上更优
- Real-event validation 下，当前 conservative decision 仍倾向 Phase 5
- 这不是简单的“谁赢谁输”，而是不同证据维度下的不同判断

### 9.4 文字要求

文字必须说明：

- 哪些数字是 frozen benchmark
- 哪些结果是最终 summary 所采用
- 哪些结论只是当前版本下的保守判断
- 不能把局部优势直接说成全面胜利

---

## 10. 真实事件验证与证据边界区块要求

这是网页中最关键的“可信度防护栏”。

### 10.1 必须单独成区块
不能把 real-event 只塞进结果区的一小部分。  
必须单独讲，因为这关系到项目可信度边界。

### 10.2 必须说明的内容

至少包括：

- 当前真实事件验证是怎样构建的
- 什么是 evidence level
- 当前证据等级主要到哪里
- 什么能说，什么不能说
- 为什么 real-event 结果要保守解读

### 10.3 必须展示的结论类型

建议分三层写：

1. **可以支持的结论**
   - 当前 default path 在保守真实事件决策规则下更稳
2. **可以支持但要保留条件的结论**
   - Phase 6 在某些 proxy 指标上更优
3. **当前不能宣称的结论**
   - 不能说模型已达到全网真实预测精度
   - 不能说系统已经生产可部署

### 10.4 失败案例要求

必须展示至少一个失败案例或限制案例，说明：
- timing error
- proxy reference 的局限
- 事件覆盖不足
- 证据等级有限

---

## 11. 工程实现与仓库体系区块要求

这个区块主要面向工程型读者与接手开发者。

### 11.1 必须说明的内容

至少包括：

- 仓库是阶段化建设的
- 每个阶段都留下了 completion/handoff/docs
- 当前 final config bundle 在哪里
- 当前 final pipeline / CLI 在哪里
- 当前主要 docs、reports、configs、src 模块怎么对应

### 11.2 不需要展示太多细节
这个区块不应变成文件树全文粘贴。  
应以“模块职责”方式介绍：

- `configs/`
- `src/gic/pipelines/`
- `src/gic/models/`
- `src/gic/eval/`
- `src/gic/visualization/`
- `reports/`
- `docs/`

### 11.3 必须体现的工程价值

要展示：
- 项目不只是模型，而是完整研究系统
- 有 CLI、有 config、有 final pipeline、有 reports、有 handoff docs
- 具备可复现、可交接、可扩展能力

---

## 12. 关键 trade-off 与设计取舍区块要求

这是网页中最能体现“工程思维”和“研究取舍”的部分。  
建议单独成区块，不要只散落在其他地方。

### 12.1 必须展示的 trade-off

至少包括：

1. **Graph baseline vs non-graph baseline**
   - 为什么 Phase 4 没有直接成为主线

2. **Physics-informed main model vs 纯图基线**
   - 为什么 Phase 5 成为核心主方法

3. **Phase 5 default vs Phase 6 KG feature_only**
   - 为什么后者保留为可选而非默认

4. **Synthetic 优势 vs Real-event 保守决策**
   - 为什么不能只看单一 benchmark

5. **继续扩模型 vs 先冻结系统**
   - 为什么 Phase 8 先做整合与复现

### 12.2 推荐写法

每个 trade-off 建议都用以下结构：

- 问题：我们面临什么选择
- 观察：实验/验证给出了什么信号
- 取舍：最终如何决策
- 原因：为什么这么做更稳妥
- 后果：这一取舍带来了什么代价 / 保留了什么扩展空间

### 12.3 这是网页的“深度价值区”
这个区块特别适合展示项目的工程成熟度。  
不是只讲结果，而是讲“为什么最后的系统长成今天这样”。

---

## 13. 当前限制与未来工作区块要求

这个区块必须真实且具体。

### 13.1 当前限制至少包括

- real-event evidence 仍然保守且不完整
- benchmark 规模仍受限
- proxy reference 不能当作 full-grid truth
- Phase 6 虽有优势，但证据不足以升为默认
- 系统不能宣称 production-ready

### 13.2 未来工作至少包括

建议按优先级写：

1. 扩大 real-event coverage
2. 增强真实证据等级
3. 继续保留 frozen baselines，不轻易破坏默认路径
4. 在更复杂条件下重新评估 KG relation / rule 路径
5. 在更广泛噪声和真实驱动下重看 signal_on / residual_mode

### 13.3 文字要求

必须体现：
- 项目已经完成一个完整版本
- 未来工作不是“补漏洞”，而是下一轮增强
- 不要把 future work 写成空泛展望

---

## 14. 关于文字写作的总体要求

虽然本文件不规定视觉风格，但对文字内容有明确要求。

### 14.1 文字类型建议
网页应至少混合使用以下几类文字：

- **一句话定位**
- **简短解释性段落**
- **对比说明文字**
- **trade-off 说明块**
- **限制提醒文字**
- **最终结论摘要**

### 14.2 写作原则

所有文字必须遵守：

- 真实
- 保守
- 可核对
- 有层次
- 不夸张
- 不营销化

### 14.3 禁止的写法

禁止：

- 把 proxy 验证说成真实全网精度
- 把 optional 路径写成失败尝试
- 只讲“成功”，不讲限制
- 只讲模型名，不讲为什么这么设计
- 只贴数字，不解释数字意味着什么

---

## 15. 推荐的网页叙事顺序

建议整体叙事按以下顺序展开：

1. 项目是什么
2. 问题为什么难
3. 项目技术路线是什么
4. 项目如何从 Phase 0 演进到 Phase 8
5. 当前系统默认路径是什么
6. 可选 KG 路径是什么，为什么保留
7. 结果怎么读
8. 真实证据边界在哪里
9. 工程上做成了什么
10. 当前限制与后续工作是什么

这个顺序的优点是：
- 先让读者理解问题
- 再理解设计
- 再看结果
- 最后理解边界和未来方向

如果后续真要做静态网页，至少应准备以下内容资产：

### 16.1 必备文案资产
- 项目简介
- 背景与动机说明
- 技术路线说明
- 阶段演进说明
- 默认 / 可选路径说明
- 结果解读文字
- 真实事件边界说明
- trade-off 说明
- 限制与 future work

### 16.2 必备图表/信息资产
- 总体技术路线图
- Phase 时间线
- 默认路径 vs KG 路径对比表
- synthetic / real-event 结果表
- evidence boundary 说明块
- 失败案例展示块
- final system module overview

### 16.3 必备链接/引用资产
- GitHub 仓库入口
- 关键文档入口
- reproduction guide
- final results summary
- final limitations
- final handoff summary

## 17. 最终结论

这个静态网页不应只是“项目介绍页”，而应是一个面向研究展示、工程交接和成果说明的**内容化入口**。  
它需要同时完成三件事：

1. 让别人快速理解项目做了什么；
2. 让别人清楚知道当前默认方案与可选方案的区别；
3. 让别人明白这个项目当前最重要的成果与最重要的边界分别是什么。

如果网页只展示结果、不展示取舍，那它会显得浅。  
如果网页只展示工程、不展示结论，那它会显得散。  
如果网页不展示边界，那它会显得不可信。

因此，最终网页内容组织必须围绕以下核心原则：

- **结果要展示**
- **取舍要解释**
- **边界要明确**
- **演进过程要可追溯**
- **工程思维要能被看见**
