# GIC 项目阶段计划（Phase Roadmap）

## 1. 文档定位与使用方式

本文档用于把整个 GIC 网络级态势重建项目拆解为一组**可执行、可验收、可回溯**的阶段，并作为后续为每个阶段编写详细开发 plan 的总索引文档。

本文件与以下文档共同构成项目参考基线：

- `GIC_project_planning_and_technical_route_v1.md`
- 本文档：`GIC_project_phase_roadmap_v1.md`

后续开发原则如下：

1. 本文档负责定义**阶段边界、目标、输入、输出、验收标准、风险与依赖关系**。
2. 每个阶段后续都将单独生成一份更细的 `phase_X_detailed_plan.md`，供 Codex 严格遵循。
3. Codex 在任意阶段开始前，必须同时参考：
   - 总体技术路线文档；
   - 本阶段路线图文档；
   - 当前阶段的详细 plan。
4. 如果某个阶段执行时与总体技术路线文档出现冲突，以**总体技术路线文档**为上位约束；如果与本阶段详细 plan 冲突，以**本阶段详细 plan**为直接执行约束。
5. 所有阶段都必须保留：
   - 可复现实验；
   - 明确的输入/输出接口；
   - 中间结果存档；
   - 失败分析；
   - 版本化配置。

---

## 2. 项目阶段划分总览

整个项目建议拆分为 **9 个阶段**。这样划分的原因是：

- 足够细，便于 Codex 严格执行；
- 又不会过碎，避免阶段之间边界不清；
- 能覆盖从“环境与数据”到“模型主干”、“KG 增强”、“真实验证”、“文档固化”的完整流程。

阶段列表如下：

- **Phase 0**：项目基线固化与工程骨架初始化
- **Phase 1**：数据资源接入与标准化数据管线建设
- **Phase 2**：GIC 物理仿真与合成标签生成框架
- **Phase 3**：前端信号处理与 quasi-DC 成分提取模块
- **Phase 4**：图结构建模与基础 GNN 重建基线
- **Phase 5**：Physics-informed 时空 GNN 主模型
- **Phase 6**：知识图谱（KG）建模与图学习增强
- **Phase 7**：真实事件驱动验证与泛化评估
- **Phase 8**：系统整合、可视化、复现与最终固化

从依赖关系看：

- Phase 0 是所有阶段前置；
- Phase 1 和 Phase 2 是主数据基础；
- Phase 3、4、5 是核心算法主线；
- Phase 6 在 Phase 5 基本稳定后推进；
- Phase 7 建立真实场景可信度；
- Phase 8 做交付级整合。

## 4. 阶段设计原则

在进入具体阶段前，先固定以下原则，后续每一阶段都必须遵守。

### 4.1 先可运行，再追求复杂度

每个阶段都优先交付：

- 最小可运行原型；
- 可验证的结果；
- 清晰的接口和数据流。

禁止一开始就写过度复杂、难以维护的“大而全”实现。

### 4.2 先基线，后增强

每个阶段都必须先建立 baseline，再做增强。  
例如：

- 先有简化物理 solver，再加复杂参数化；
- 先有基础 GCN/GAT，再做 physics-informed hetero temporal GNN；
- 先有轻量 KG schema，再做 relation-aware 融合。

### 4.3 接口优先于内部细节

后续阶段之间的衔接主要依赖接口，因此每个阶段都要优先定义：

- 输入格式；
- 输出格式；
- 配置结构；
- 日志与缓存；
- 评估结果的保存方式。

### 4.4 每个阶段都必须可验收

不允许出现“做了一些东西但不知道算不算完成”的状态。  
每个阶段都必须有明确的：

- 完成标准；
- 最低验收标准；
- 失败标准；
- 暂缓升级条件。

### 4.5 所有阶段都必须服务于最终两条主线

整个项目只有两条真正的主目标线：

1. **网络级 GIC 态势重建**
2. **物理 + 图学习 + KG 的协同建模**

所有阶段都要解释自己如何支持这两条主线。

## 5. Phase 0：项目基线固化与工程骨架初始化

## 5.1 阶段定位

Phase 0 不是算法阶段，而是整个项目的**工程起点与约束固化阶段**。  
目标是形成一套后续所有阶段都遵守的仓库结构、配置方式、依赖管理、实验记录和文档组织方式。

没有这个阶段，后续 Codex 很容易在不同阶段生成风格不统一、目录混乱、接口不一致的代码。

## 5.2 阶段目标

建立一个适合长期演进的项目骨架，使后续每个阶段都能在统一规范下实现、测试和迭代。

## 5.3 本阶段要做的事情

### 5.3.1 建立仓库目录骨架

至少应包含以下顶层目录：

- `configs/`
- `data/`
- `docs/`
- `notebooks/`
- `scripts/`
- `src/`
- `tests/`
- `artifacts/`
- `logs/`
- `reports/`

并在 `src/` 内预留模块目录，例如：

- `src/data/`
- `src/signal/`
- `src/physics/`
- `src/graph/`
- `src/models/`
- `src/kg/`
- `src/eval/`
- `src/utils/`
- `src/visualization/`

### 5.3.2 固定配置体系

定义统一配置机制，例如：

- YAML 配置为主；
- 每个阶段一个主配置；
- 支持从 CLI 覆盖关键参数；
- 训练、数据、物理、图、KG、评估配置分离。

### 5.3.3 固定实验记录规则

明确以下内容：

- 每次实验必须有唯一 run_id；
- 每次实验必须保存：
  - 配置快照；
  - git commit 哈希；
  - 日志；
  - 指标；
  - 产出模型；
  - 中间图表；
- 报告路径规范。

### 5.3.4 固定代码质量约束

至少确定：

- Python 版本；
- 包管理方式；
- 格式化工具；
- lint 工具；
- 单元测试框架；
- 类型注解规范；
- 日志规范。

### 5.3.5 固定文档索引结构

在 `docs/` 下建立：

- `docs/roadmap/`
- `docs/phases/`
- `docs/data/`
- `docs/models/`
- `docs/evaluation/`
- `docs/decisions/`

为后续阶段文档预留位置。

## 5.4 本阶段产出物

- 初始仓库目录结构
- 项目 README 初稿
- 开发环境配置文件
- 统一配置模板
- 日志与 artifacts 规范说明
- 阶段文档目录骨架
- 基础测试脚手架

## 5.5 阶段完成标准

满足以下条件即可视为完成：

1. 仓库可成功安装依赖；
2. 至少有一个最小 CLI 能读取配置并输出 run_id；
3. 测试命令可运行；
4. 文档目录齐全；
5. 后续阶段可以在此骨架上直接继续开发。

## 5.6 本阶段与总体目标的关系

Phase 0 不直接提升模型效果，但它决定后续项目是否能稳定演进。  
它是所有阶段的基础设施层。

## 6. Phase 1：数据资源接入与标准化数据管线建设

## 6.1 阶段定位

这是项目的数据底座阶段。目标是把目前规划文档中确定准备使用的数据源、测试系统和原始文件统一纳入一个**标准化数据访问层**。

## 6.2 阶段目标

建立统一的数据接入、缓存、标准化、元数据管理和切分机制，使后续物理仿真、信号处理、图建模和训练都使用同一套数据接口。

## 6.3 本阶段要做的事情

### 6.3.1 接入准备使用的核心数据与测试系统

优先纳入以下类型资源：

- 电网测试系统：
  - MATPOWER case118
  - UIUC 150-bus / TAMU 相关测试系统
- 地磁驱动数据：
  - SuperMAG
  - INTERMAGNET
  - NOAA 指数或地电场产品
- 可用的真实 GIC 观测样例或论文附带数据（若可获取）
- 本地手工整理的设备与参数文件

### 6.3.2 定义统一数据对象

应设计标准对象来表示：

- 电网图静态信息
- 时序驱动信息
- 传感器观测信息
- 物理参数
- 事件级元数据
- 数据分片索引

### 6.3.3 完成原始数据到标准格式转换

包括但不限于：

- 文件读取器
- 时间戳对齐
- 单位统一
- 缺失值标记
- 区域坐标标准化
- 网络节点与设备 id 对齐

### 6.3.4 建立数据注册表

每个数据源都要有注册信息，包括：

- 名称
- 来源
- 原始路径
- 许可说明
- 时间范围
- 空间范围
- 字段含义
- 是否已纳入训练
- 是否仅作验证

### 6.3.5 明确数据切分策略

包括：

- 训练/验证/测试切分；
- 按 storm event 切分；
- 按稀疏率场景切分；
- 按噪声强度场景切分；
- 为后续 cross-event / cross-topology 泛化预留方案。

## 6.4 本阶段产出物

- 标准化数据访问模块
- 各类 parser/loader
- 数据注册表
- 数据 schema 文档
- 至少一套可读入的测试系统与时序驱动样例
- 数据质量检查脚本

## 6.5 阶段完成标准

1. 可以通过统一接口加载至少一个电网测试系统；
2. 可以加载至少一种时序地磁/地电场驱动；
3. 可以输出标准化后的中间文件；
4. 可以通过脚本检查数据完整性；
5. 后续阶段不再直接依赖原始杂乱文件。

## 6.6 本阶段关键风险

- 不同数据源时间粒度不一致；
- 测试系统参数缺失；
- 真实 GIC 数据难获取；
- 地理坐标和拓扑节点映射复杂。

## 6.7 本阶段与总体目标的关系

Phase 1 决定项目的数据基础是否可靠，是所有后续算法阶段的前置条件。

---

## 7. Phase 2：GIC 物理仿真与合成标签生成框架

## 7.1 阶段定位

这是核心物理基础阶段。目标是把 GIC 重建问题中最关键的“真值来源”建立起来：  
即，基于电网拓扑、线路参数、接地参数和地电场驱动，生成 transformer-level / line-level / node-level 的 GIC baseline 与标签。

## 7.2 阶段目标

建立一个模块化、可扩展、可测试的 GIC forward solver，并在其基础上生成用于训练和评估的合成标签数据。

## 7.3 本阶段要做的事情

### 7.3.1 实现基础 GIC 计算流程

包括：

- 线路切向地电场积分
- 线电压等效表示
- Norton 等效电流注入
- DC 网络求解
- 设备层 GIC 计算

### 7.3.2 设计参数接口

solver 至少应支持：

- 线路长度与方向
- 电阻/导纳
- 接地参数
- 是否串补
- 变压器类型或等效通路参数
- 区域电场输入

### 7.3.3 支持多种驱动模式

至少支持：

- 理想均匀地电场输入
- 真实事件插值后的空间场输入
- 人工构造场景
- 参数扰动场景

### 7.3.4 生成多层级标签

建议生成：

- transformer-level GIC
- line-level induced quantities
- node-level electrical quantities
- region-level aggregated risk score
- hotspot mask

### 7.3.5 生成多场景训练样本

包括：

- 不同 storm 强度
- 不同空间场形状
- 不同参数扰动
- 不同观测稀疏率
- 不同噪声强度

## 7.4 本阶段产出物

- GIC forward solver
- solver 单元测试
- 合成数据生成脚本
- 训练标签文件
- 场景配置模板
- baseline 可视化结果

## 7.5 阶段完成标准

1. solver 可在至少一个测试系统上运行；
2. 能输出稳定的 GIC 数值结果；
3. 能生成批量样本；
4. 结果可被下游模型读取；
5. 可以通过人工 sanity check 验证基础合理性。

## 7.6 本阶段关键风险

- 测试系统缺少做 GIC 所需的部分参数；
- 需要对变压器和接地做合理简化；
- 真实电场分布映射到线路时存在误差；
- 不同论文公式实现细节可能不完全一致。

## 7.7 本阶段与总体目标的关系

这是项目标签来源的核心，没有 Phase 2，后续模型训练无法系统推进。

## 8. Phase 3：前端信号处理与 quasi-DC 成分提取模块

## 8.1 阶段定位

这是时序前端阶段，目标是把原始传感器或高频时序中的有效 GIC 相关低频成分提取出来，并形成统一的时序特征表示。

## 8.2 阶段目标

实现可比较、可替换的前端信号处理模块，为后续图模型提供更稳健的输入。

## 8.3 本阶段要做的事情

### 8.3.1 建立前端方法集合

至少包括三类：

- FastICA / 盲源分离类
- 鲁棒低频滤波 / 趋势提取类
- 自适应稀疏表示 / 字典学习类

### 8.3.2 建立统一输入输出接口

输入：

- 多通道传感器时序
- 原始或合成高频信号
- 噪声配置

输出：

- quasi-DC 成分
- 降噪后特征
- 质量指标

### 8.3.3 设计对比实验

比较维度至少包括：

- 信噪比改善
- 低频保真度
- 峰值保持
- 对后续重建性能的影响
- 计算开销

### 8.3.4 建立前端模块选择策略

最终不必只保留一个方法，应允许：

- 固定单方法；
- 配置切换；
- 融合输出；
- 自动选择。

## 8.4 本阶段产出物

- 信号处理模块
- 多方法对比脚本
- 前端评估指标
- 降噪后样例
- 统一时序特征导出接口

## 8.5 阶段完成标准

1. 至少三种前端方法可运行；
2. 可在统一数据接口上比较；
3. 可输出供图模型使用的标准特征；
4. 有明确推荐的默认前端方案。

## 8.6 本阶段关键风险

- FastICA 假设不一定稳定成立；
- 真实噪声模式与合成噪声差异大；
- 信号前端改善不一定直接转化为最终重建提升。

## 8.7 本阶段与总体目标的关系

这一阶段提升输入质量，降低观测噪声对后续 GNN 的干扰。

---

## 9. Phase 4：图结构建模与基础 GNN 重建基线

## 9.1 阶段定位

这是第一阶段真正的“学习式重建”实现。目标是先建立一个不依赖复杂物理约束的基础 GNN baseline，验证图建模这条主线本身是否有效。

## 9.2 阶段目标

构建标准图数据表示与基础 GNN 基线模型，实现从稀疏观测到网络级 GIC map 的初步重建。

## 9.3 本阶段要做的事情

### 9.3.1 设计图结构表达

确定：

- 节点类型：bus / transformer / substation / sensor
- 边类型：electrical connectivity / observation linkage / optional geographic relation
- 节点特征与边特征定义
- 时序窗口编码方式

### 9.3.2 实现基础图数据集构造器

支持：

- 单时刻图
- 滑动窗口图
- 稀疏观测 mask
- 多目标标签加载

### 9.3.3 实现 baseline 模型

建议至少实现：

- MLP baseline
- GCN baseline
- GraphSAGE baseline
- GAT baseline

### 9.3.4 建立基础训练与评估流程

包括：

- 训练脚本
- 验证脚本
- 指标统计
- checkpoint 保存
- 结果可视化

### 9.3.5 做基础 ablation

包括：

- 无图结构 vs 有图结构
- 不同稀疏率
- 不同噪声强度
- 不同邻域半径
- 不同输入特征组合

## 9.4 本阶段产出物

- 图数据构造模块
- 基础 GNN 模型
- baseline 训练脚本
- baseline 评估报告
- 第一版可视化重建结果

## 9.5 阶段完成标准

1. 可以在合成数据上训练并收敛；
2. GNN 明显优于非图 baseline；
3. 可输出网络级重建结果；
4. 可生成阶段性实验报告。

## 9.6 本阶段关键风险

- 图结构设计不合理会导致效果不稳定；
- 节点/边定义过于粗糙会弱化物理局域性；
- 早期指标可能对热点设备不敏感。

## 9.7 本阶段与总体目标的关系

Phase 4 用于验证“图建模是否值得继续深化”。这是迈向 physics-informed GNN 的必要前置阶段。

---

## 10. Phase 5：Physics-informed 时空 GNN 主模型

## 10.1 阶段定位

这是整个项目最核心的算法阶段。目标是把物理 baseline、时序信息、图结构和稀疏观测整合成主模型。

## 10.2 阶段目标

实现一个可训练、可解释、可评估的 physics-informed temporal GNN，作为项目主方法。

## 10.3 本阶段要做的事情

### 10.3.1 将物理求解结果纳入模型输入或约束

方式可以包括：

- 把物理 baseline 当作节点特征；
- 让模型学习 residual correction；
- 在 loss 中加入 physics consistency penalty；
- 在图边权中引入物理敏感度信息。

### 10.3.2 引入时序建模

可选方案包括：

- TCN + GNN
- Temporal Graph Network
- RNN/GRU + GNN
- Graph Transformer 风格时空编码

### 10.3.3 设计多任务输出头

建议包括：

- GIC regression head
- hotspot classification head
- risk ranking head
- uncertainty head

### 10.3.4 设计综合损失函数

至少考虑：

- MSE / MAE
- hotspot 分类损失
- ranking loss
- physics penalty
- spatial smoothness penalty
- temporal consistency penalty

### 10.3.5 做系统 ablation 与主实验

包括：

- baseline GNN vs physics-informed GNN
- 有无前端去噪
- 有无物理 baseline
- 有无时序模块
- 不同稀疏率
- 不同拓扑与 storm 场景

## 10.4 本阶段产出物

- 主模型代码
- 多任务训练脚本
- 完整评估管线
- ablation 结果
- 主实验报告

## 10.5 阶段完成标准

1. 主模型优于 Phase 4 基线；
2. 在中高稀疏率下仍有稳定表现；
3. 能同时输出回归结果与风险识别结果；
4. 有清晰的误差与泛化分析。

## 10.6 本阶段关键风险

- 模型复杂度过高导致训练不稳定；
- 物理约束加得过重，反而降低拟合能力；
- 多任务之间存在冲突；
- 指标提升可能只体现在均值而非关键风险点。

## 10.7 本阶段与总体目标的关系

这是网络级 GIC 重建主线的中心阶段，决定项目的核心算法贡献。

## 11. Phase 6：知识图谱（KG）建模与图学习增强

## 11.1 阶段定位

Phase 6 不是替代主模型，而是引入**语义关系、规则先验、异构实体组织能力**，增强模型的结构表达与解释能力。

## 11.2 阶段目标

建立与项目匹配的 KG schema、实体关系层和 KG-to-model 融合机制，使 KG 成为主项目目标的一部分，而不是附属说明。

## 11.3 本阶段要做的事情

### 11.3.1 设计 KG schema

实体建议至少包括：

- Grid
- Region
- Substation
- Bus
- Transformer
- TransmissionLine
- Sensor
- MagnetometerStation
- StormEvent
- GeoelectricRegion
- MaterializedObservation
- RiskState

关系建议至少包括：

- connected_to
- located_in
- observes
- influenced_by
- grounded_via
- belongs_to_event
- has_transformer_type
- has_voltage_level
- has_sensor
- adjacent_to_region
- constrained_by_rule

### 11.3.2 构建实体与关系抽取流程

从现有数据与配置中抽取：

- 拓扑关系
- 地理关系
- 设备属性
- 观测关系
- 规则关系
- 事件关系

### 11.3.3 设计 KG 与主模型融合策略

可选路径包括：

- KG 生成 node/edge attributes
- relation embedding 注入 hetero-GNN
- rule-based constraint feature
- 缺失属性补全
- 异常关系检测

### 11.3.4 建立 KG 评价方式

不只看“是否构建成功”，还要看：

- 是否提升主模型性能；
- 是否改善缺失属性场景；
- 是否增强可解释性；
- 是否支持知识查询与错误检查。

## 11.4 本阶段产出物

- KG schema 文档
- KG 构建脚本
- KG 数据文件
- KG 融合模块
- KG 增益实验报告

## 11.5 阶段完成标准

1. KG 能从现有数据自动或半自动构建；
2. KG 至少以一种方式进入主模型；
3. 在至少一个场景下带来可衡量收益；
4. 可给出一组基于 KG 的语义解释样例。

## 11.6 本阶段关键风险

- KG 设计过于庞大导致开发效率低；
- 语义层复杂但对预测性能帮助有限；
- 数据源不完整使关系抽取困难。

## 11.7 本阶段与总体目标的关系

Phase 6 负责把“KG 也是目标的一部分”落到实处，增强结构表达、语义组织和可解释能力。

---

## 12. Phase 7：真实事件驱动验证与泛化评估

## 12.1 阶段定位

这是项目可信度建设阶段。目标是让系统不只在合成数据上表现良好，还能在真实地磁事件、真实驱动或局部真实观测下展示可用性。

## 12.2 阶段目标

完成跨事件、跨输入来源、跨噪声强度和可用真实观测条件下的综合验证。

## 12.3 本阶段要做的事情

### 12.3.1 接入真实 storm 驱动

选取若干具有代表性的 geomagnetic storm 事件，构建真实驱动输入集。

### 12.3.2 做事件级验证

包括：

- 峰值时段对齐
- 相对风险排序
- 局部测点趋势相关性
- 热点区域识别

### 12.3.3 做泛化实验

包括：

- cross-event
- cross-sparsity
- cross-noise
- cross-topology（若可行）

### 12.3.4 做鲁棒性评估

包括：

- 传感器失效
- 参数扰动
- 时钟偏移
- 缺失属性
- 驱动场偏差

## 12.4 本阶段产出物

- 真实事件测试集
- 泛化评估脚本
- 鲁棒性报告
- 外部验证报告

## 12.5 阶段完成标准

1. 至少完成一组真实事件驱动验证；
2. 有跨场景泛化结果；
3. 能说明系统在真实条件下的边界与局限；
4. 有清晰的可信度陈述。

## 12.6 本阶段关键风险

- 真实 GIC 真值稀缺；
- 真实驱动数据与测试系统映射有偏差；
- 合成与真实之间 domain gap 明显。

## 12.7 本阶段与总体目标的关系

Phase 7 决定项目是否具有“从研究原型迈向可信方法”的资格。

---

## 13. Phase 8：系统整合、可视化、复现与最终固化

## 13.1 阶段定位

这是最终交付阶段。目标是把前面所有分散成果整合成可重复运行、可演示、可持续维护的系统版本。

## 13.2 阶段目标

完成系统级整合、可视化输出、实验复现、最终文档和项目固化。

## 13.3 本阶段要做的事情

### 13.3.1 整合统一 pipeline

将数据、物理、前端、主模型、KG、评估串联为一个端到端流程。

### 13.3.2 提供可视化输出

至少包括：

- GIC map 可视化
- 热点设备榜单
- 时序峰值图
- 误差分析图
- KG 关系示意或查询示例

### 13.3.3 清理与重构代码

包括：

- 删除废弃脚本
- 合并重复模块
- 稳定接口
- 增补测试

### 13.3.4 固化文档与复现流程

输出：

- 最终 README
- 运行手册
- 复现实验说明
- 配置模板
- 阶段总结与 decision log

### 13.3.5 准备后续扩展方向

例如：

- 更大规模测试系统
- 更复杂真实数据接入
- 更强时空模型
- 更完整 KG 推理
- 在线推断或交互式可视化

## 13.4 本阶段产出物

- 复现实验脚本

## 13.5 阶段完成标准

1. 可以从原始输入跑到最终输出；
2. 至少一条标准实验路径可完整复现；
3. 文档足以支撑第三方阅读与继续开发；
4. 所有阶段成果被统一纳入主仓库。

## 13.6 本阶段关键风险

- 前期代码风格不统一导致整合成本高；
- 中间模块接口变动过多；
- 文档与代码不同步。

## 13.7 本阶段与总体目标的关系

这是把研究项目转化为长期可维护资产的最后一步。

## 14. 阶段优先级与执行顺序建议

建议严格按以下顺序推进：

1. Phase 0
2. Phase 1
3. Phase 2
4. Phase 3 与 Phase 4 可部分并行
5. Phase 5
6. Phase 6
7. Phase 7
8. Phase 8

### 14.1 为什么不建议过早做 KG

KG 很重要，但不应早于主重建链路。原因是：

- 如果主数据与主模型还没跑通，KG 增强没有明确落点；
- KG 设计过早会消耗大量建模精力；
- KG 的价值更依赖已有系统接口。

因此 KG 作为目标的一部分是正确的，但在执行顺序上应位于主模型稳定之后。

## 17. 阶段失败与暂停条件

为了避免项目无限膨胀，也需要定义阶段失败或暂停条件。

### 17.1 可接受失败

以下情况可以接受，只要文档化清楚：

- 某种前端方法效果不好；
- 某个 KG 融合策略没有收益；
- 某一类真实数据无法稳定接入；
- 某种模型结构训练不稳定。

### 17.2 不可接受失败

以下情况不可接受：

- 阶段产出没有接口；
- 结果不可复现；
- 配置和代码不一致；
- 中间结果没有保存；
- 文档缺失导致下阶段无法继续。

### 17.3 暂停升级条件

若某阶段出现以下情况，不应立刻进入下一阶段：

- 关键 baseline 尚未建立；
- 数据格式仍频繁变化；
- 主指标未达到最低门槛；
- 测试系统结果存在明显物理异常且未解释。

## 19. 本文档的最终用途

## 20. 最终结论

整个项目最合理的推进方式不是一次性实现所有设想，而是按照本文定义的阶段逐步推进：

- 先固定工程骨架；
- 再固化数据；
- 再建立物理标签来源；
- 再做信号前端与图重建基线；
- 再做 physics-informed 主模型；
- 再把 KG 作为正式目标纳入；
- 最后做真实验证与系统整合。

这样拆分后，项目具有以下优点：

1. 每个阶段都有明确边界；
2. 每个阶段都能单独形成可验收成果；
3. Codex 可以被严格约束；
4. 总体风险可控；
5. 后续论文、报告、演示与工程实现都更容易组织。
